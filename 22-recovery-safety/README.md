# 22 回退版本的安全判斷完整教學

> 第 21 章教你「有哪些指令可以退回版本」。  
> 第 22 章要教你更重要的事：什麼叫安全，什麼時候該用哪種退法，怎麼退才不容易把自己或隊友弄亂。

## 先講最重要的觀念：Git 裡的「安全」是什麼

很多新手以為「安全」就是：

1. 不會出錯。
2. 不會刪到東西。
3. 指令看起來比較溫和。

其實 Git 裡更實際的安全，通常是下面這幾件事：

| 安全代表什麼 | 說明 |
| --- | --- |
| 不亂改共享歷史 | 已經 push、而且別人也可能拉走的歷史，不要輕易重寫 |
| 不意外丟掉本機修改 | 你至少要知道哪些檔案會消失、哪些提交會退掉 |
| 出事時比較容易救 | 比如 `revert`、`stash`、`reflog` 這類做法，通常可回復性比較高 |
| 影響範圍清楚 | 你知道影響的是工作區、暫存區、提交歷史，還是遠端 |
| 執行前有先確認目標 | 你很清楚自己要退到哪個 commit、哪個 branch、哪個遠端狀態 |

所以：

- 安全，不代表完全沒有風險。
- 安全，代表你知道風險在哪裡，而且知道怎麼降低它。

## 第 21 章有沒有不好的地方

第 21 章不是錯，它是速查表。  
但速查表的缺點就是：很快、很直接，也很容易讓人跳過判斷。

| 第 21 章的寫法 | 容易讓人忽略什麼 |
| --- | --- |
| `git reset --hard` 出現很多次 | 新手可能直接照抄，沒先確認未提交修改會不會被清掉 |
| `git push --force` 直接列出來 | 容易讓人以為這是一般操作，而不是高風險操作 |
| `origin/main` 當範例 | 新手可能忘了自己的專案不一定叫 `main` |
| `git clean -fd` 直接給答案 | 可能沒先檢查未追蹤檔案裡有沒有重要內容 |
| 直接建議重新 clone | 有時最快，但也可能讓你略過救援和備份步驟 |

所以更準確的定位是：

1. 第 21 章是急救卡。
2. 第 22 章是判斷課。
3. 真的要操作時，先用第 22 章判斷，再回第 21 章查指令。

## 先學會看風險：哪些指令比較安全

| 指令 | 安全程度 | 為什麼 |
| --- | --- | --- |
| `git status` | 很安全 | 只看狀態，不改任何內容 |
| `git log --oneline` | 很安全 | 只看歷史 |
| `git diff` | 很安全 | 只看差異 |
| `git stash` | 相對安全 | 先把修改收起來，不是直接刪掉 |
| `git revert` | 相對安全 | 不重寫歷史，而是新增一筆反向提交 |
| `git reset --soft` | 中低風險 | 退掉 commit，但修改還在 |
| `git reset` | 中風險 | 退掉 commit，修改回工作區 |
| `git reset --hard` | 高風險 | commit、暫存區、工作區一起退 |
| `git clean -fd` | 高風險 | 未追蹤檔案會直接刪掉 |
| `git push --force` | 很高風險 | 可能改亂遠端共享歷史 |

這裡最重要的不是背表格，而是理解一個方向：

- 越會改寫歷史的，通常越危險。
- 越會直接刪掉本機內容的，通常越危險。
- 越會影響遠端和隊友的，通常越危險。

## 每次回退前，固定先問自己 4 個問題

### 1. 你要取消的是哪一層

| 你想取消什麼 | 常見工具 |
| --- | --- |
| 工作區還沒 commit 的修改 | `git restore`、`git stash` |
| 剛剛那筆 commit，但修改想保留 | `git reset --soft`、`git reset` |
| 剛剛那筆 commit，而且修改也不要了 | `git reset --hard` |
| 已經 push 的提交 | `git revert`，或非常小心地 force push |

### 2. 那筆提交有沒有 push 過

| 狀態 | 判斷重點 |
| --- | --- |
| 還沒 push | 你通常可以比較自由地整理歷史 |
| 已經 push | 要先考慮遠端是否已被別人使用 |

### 3. 這是不是共享分支

| 分支類型 | 安全原則 |
| --- | --- |
| 你自己的功能分支 | 可以考慮改歷史，但還是要小心 |
| 團隊共同使用的 `main`、`master`、共享分支 | 優先考慮 `revert`，不要直接重寫 |

### 4. 你現在有沒有未提交修改

如果有，那你最先該擔心的不是怎麼退回版本，而是：
「這些修改會不會先被我自己弄丟？」

## 危險操作前，標準檢查流程

先跑這四個：

```bash
git status
git log --oneline --decorate -5
git diff
git diff --staged
```

| 指令 | 你在確認什麼 |
| --- | --- |
| `git status` | 目前乾不乾淨，有沒有未提交修改 |
| `git log --oneline --decorate -5` | 你現在在哪裡，上一筆是誰，分支指到哪裡 |
| `git diff` | 工作區有什麼沒存進 commit 的內容 |
| `git diff --staged` | 暫存區準備提交的是什麼 |

這四步很重要，因為很多 Git 事故不是指令不會用，而是「沒先看清楚自己現在在哪裡」。

## 如果你怕退錯，先做保險

### 做法一：先開一個備份分支

```bash
git branch backup/before-recovery
```

適合：

- 你準備做 `reset`
- 你準備做大範圍整理
- 你怕自己等等找不到目前這個位置

### 做法二：先把工作區修改收起來

```bash
git stash -u
```

適合：

- 你工作做到一半
- 你還不確定這些修改要不要留
- 你等等要做 `reset --hard` 或切 branch

### 做法三：要刪未追蹤檔案前，先試跑

```bash
git clean -nd
```

這一步只看不刪，很值得養成習慣。

## `reset` 到底差在哪裡

這是第 22 章最重要的核心之一。

| 指令 | 會動到 commit 歷史 | 會動到暫存區 | 會動到工作區 | 適合什麼情況 |
| --- | --- | --- | --- | --- |
| `git reset --soft HEAD~1` | 會 | 會保留成 staged | 不動 | 你只是想重做上一筆 commit |
| `git reset HEAD~1` | 會 | 會重設 | 修改留在工作區 | 你想拆開重做這次提交 |
| `git reset --hard HEAD~1` | 會 | 會重設 | 會一起退回 | 你確定這些修改和提交都不要了 |

### 怎麼理解這三個差別

#### `--soft`

最像：
「這筆 commit 我想重做，但內容其實我還要。」

#### 預設 `reset`，也就是 mixed

最像：
「這筆 commit 我先拆掉，內容放回工作區，我自己重新整理。」

#### `--hard`

最像：
「這整段我都不要了，連工作區也一起丟掉。」

所以很多時候，真正比較安全的不是直接 `--hard`，而是先想：

1. 我只是 commit 訊息錯了？
2. 我只是想重組提交？
3. 我只是想把檔案退回工作區？

如果答案是這些，那你很可能根本不用 `--hard`。

## 完整教學情境一：還沒 commit，只是工作區改亂了

### 狀況

你改了一些檔案，但還沒 commit。  
你現在只想把某些檔案或全部修改丟掉。

### 比較安全的做法

先看：

```bash
git status
git diff
```

如果你確定某個檔案不要了：

```bash
git restore README.md
```

如果你還不確定，但想先收起來：

```bash
git stash -u
```

### 為什麼這樣比較安全

| 做法 | 安全原因 |
| --- | --- |
| `git restore <file>` | 只影響指定檔案，不會去改提交歷史 |
| `git stash -u` | 不是直接刪掉，而是先收起來，之後還能 `stash pop` |

## 完整教學情境二：已經 commit，但還沒 push，想重做上一筆

### 狀況

你已經 commit 了，但發現：

- commit 訊息打錯
- 多加了不該加的檔案
- 想把一筆 commit 拆成兩筆

### 建議做法

如果只是補檔案或改訊息：

```bash
git commit --amend
```

如果想整筆拆掉但保留內容：

```bash
git reset --soft HEAD~1
```

或：

```bash
git reset HEAD~1
```

### 為什麼這樣算安全

| 指令 | 為什麼比較安全 |
| --- | --- |
| `git commit --amend` | 只修最近一筆提交，範圍小 |
| `git reset --soft HEAD~1` | 提交消失，但內容還保留在暫存區 |
| `git reset HEAD~1` | 提交退回，但內容還留在工作區 |

### 這時通常不需要

```bash
git reset --hard HEAD~1
```

因為你的目標不是把內容丟掉，而是把提交整理好。

## 完整教學情境三：已經 commit，但還沒 push，而且你真的不要那筆內容了

### 狀況

你剛剛那筆 commit 是錯的，而且你很確定：

- 那些修改不需要保留
- 工作區也不用救

### 做法

先最後確認一次：

```bash
git status
git log --oneline -3
```

再執行：

```bash
git reset --hard HEAD~1
```

### 為什麼這還算相對安全

因為它還沒 push，影響通常只有你自己。  
但它仍然不是低風險，因為你可能把本機修改一起清掉。

### 更穩一點的版本

如果你還有一點不確定，可以先：

```bash
git branch backup/before-hard-reset
```

再做 `reset --hard`。

## 完整教學情境四：已經 push 到遠端，但這是你自己的分支

### 狀況

你已經把 commit push 上去了。  
但這個分支是你自己的功能分支，沒有別人一起用。

### 可以怎麼做

```bash
git log --oneline -3
git reset --hard HEAD~1
git push --force-with-lease
```

### 為什麼這不算最安全，但還可以接受

| 原因 | 說明 |
| --- | --- |
| 已經 push 了 | 表示你開始碰到遠端歷史風險 |
| 但分支是你自己的 | 影響範圍主要還是你自己 |
| `--force-with-lease` 比 `--force` 好一點 | 如果遠端有你沒預期的更新，它會先擋下來 |

### 注意

如果這個分支其實有人也會 checkout 或 review，那它就不再算真正的私人分支。

## 完整教學情境五：已經 push，而且是共享分支，想取消上一筆

### 狀況

你已經把內容 push 到：

- `main`
- `master`
- 團隊一起用的 branch

而且你想取消上一筆提交。

### 建議做法

```bash
git log --oneline -3
git revert HEAD
git push
```

### 為什麼這是比較安全的標準答案

| 原因 | 說明 |
| --- | --- |
| 不改寫歷史 | 舊 commit 還在 |
| 對隊友比較友善 | 別人 pull 時不會突然發現歷史被重寫 |
| 可追蹤 | 日後看紀錄，知道你是用一筆反向 commit 把它取消 |

### 什麼情況不要直接 `reset --hard` + `push --force`

幾乎所有共享分支都不建議這樣做。  
除非你的團隊流程本來就允許，而且大家都知道要怎麼處理。

## 完整教學情境六：GitHub 上是對的，本機亂掉了，想完全對齊遠端

### 狀況

你的本機現在：

- 有亂改的檔案
- 有沒追蹤的垃圾檔
- 你只想讓本機變成 GitHub 現在那版

### 比較完整的做法

先確認分支名稱：

```bash
git branch -r
```

再確認本機目前狀態：

```bash
git status
```

如果你確定本機內容不要了：

```bash
git fetch origin
git reset --hard origin/main
git clean -nd
git clean -fd
```

如果你的主要分支是 `master`，就把 `origin/main` 改成 `origin/master`。

### 為什麼這裡先 `clean -nd`

因為 `git clean -fd` 會刪掉未追蹤檔案。  
先試跑，才知道哪些東西會消失。

### 這個流程的安全前提

1. 你非常確定遠端版本是正確的。
2. 你非常確定本機修改不要了。
3. 你已經確認過未追蹤檔案裡沒有重要資料。

## 完整教學情境七：你剛剛 reset 完，才發現退錯了

### 狀況

你剛剛做了：

```bash
git reset --hard HEAD~1
```

然後馬上後悔。

### 先不要慌，先看：

```bash
git reflog
```

你會看到 HEAD 最近移動過哪些位置。  
找到你剛剛退掉之前那個位置後，可以再切回去。

例如：

```bash
git reset --hard HEAD@{1}
```

或退回某個 reflog 裡看到的 commit。

### 為什麼 `reflog` 很重要

因為它記錄的是：
「你的 HEAD 最近去過哪裡」

不是只有正式歷史。  
所以很多 `reset`、`switch`、`amend` 之後找不到的東西，常常能先靠它找到。

## `git clean` 為什麼常常被低估

很多人把注意力都放在 `reset --hard`，但其實：

```bash
git clean -fd
```

也很危險，因為它刪的是未追蹤檔案。

這包含：

- 還沒 `git add` 的新檔案
- 暫存資料夾
- 測試輸出
- 下載資料
- 你手動放進來但 Git 還沒追蹤的內容

所以更安全的順序應該是：

```bash
git clean -nd
git clean -fd
```

## `push --force` 和 `push --force-with-lease` 差在哪裡

| 指令 | 差別 |
| --- | --- |
| `git push --force` | 幾乎就是直接強蓋遠端 |
| `git push --force-with-lease` | 會先確認遠端是不是還是你以為的那個狀態 |

### 為什麼 `--force-with-lease` 比較安全

如果有人在你 force push 之前，已經先推了別的更新，  
`--force-with-lease` 比較有機會擋下來，不讓你直接蓋掉別人的工作。

它不是完全無風險，但通常是更好的預設。

## 一張真正實用的判斷表

| 你的情況 | 優先做法 | 為什麼 |
| --- | --- | --- |
| 還沒 commit，只想丟掉某個檔案修改 | `git restore <file>` | 只影響單一檔案 |
| 還沒 commit，但先不想丟 | `git stash -u` | 先收起來，之後還能拿回來 |
| 已 commit、未 push，只想重做提交 | `git commit --amend`、`git reset --soft`、`git reset` | 不必直接丟內容 |
| 已 commit、未 push，而且內容真的不要了 | `git reset --hard` | 影響主要只有自己 |
| 已 push，而且是共享分支 | `git revert` | 不重寫共享歷史 |
| 本機要和遠端完全一致 | `git fetch` + `git reset --hard` + `git clean -nd` + `git clean -fd` | 先同步，再清理本機 |
| 剛做錯 reset | `git reflog` | 先找回剛剛的位置 |

## 最後給你的安全原則

1. 已經分享出去的歷史，優先想 `revert`，不要先想 `reset --hard`。
2. 只要你還想保留內容，就先不要碰 `--hard`。
3. 做危險操作前，先看 `status`、`log`、`diff`。
4. 怕出事就先做保險：備份 branch、`stash`、`clean -nd` 試跑。
5. 看到 `push --force`，先停一下，問自己這是不是共享分支。
6. 真出事了先看 `reflog`，不要第一時間判定「完了」。

## 本章你要記住

1. Git 裡的安全，不是完全不會出事，而是你知道風險、影響範圍、和救援方式。
2. 真正的安全判斷，先看三件事：有沒有 push、是不是共享分支、未提交修改要不要留。
3. `revert` 通常比改寫共享歷史安全，`reset --soft` 和 `reset` 通常比 `reset --hard` 安全。
4. `git clean -fd` 和 `git push --force` 都不能當成一般日常指令隨手用。
5. 第 21 章背指令，第 22 章學判斷，兩章一起用才是真的會退版本。
