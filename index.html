起手式：第一次把專案推上 GitHub

最常用組合

幾乎每個人第一次都在跑這一套。先整套看過、跟著做一次就好 —— 看不懂沒關係，後面每一章其實都在拆解這幾行各自在幹嘛。

📋 一套指令：從空資料夾 → GitHub

先到 GitHub 按 New repository 開一個空倉庫（先別勾 README / .gitignore），拿到網址。接著在你的專案資料夾裡，由上往下一行一行執行：

第一次上傳的黃金 7 步
git init                                   # ① 把這個資料夾變成 Git 倉庫
# ② 建立 .gitignore，先把 node_modules/、.env 等不該上傳的擋掉
git add .                                  # ③ 把所有變更放進暫存區
git commit -m "first commit"               # ④ 建立第一個版本（快照）
git branch -M main                         # ⑤ 把主分支命名為 main
git remote add origin https://github.com/你的帳號/新倉庫.git   # ⑥ 接上 GitHub
git push -u origin main                      # ⑦ 第一次推上去
🔑
只有 2 個地方要換成你自己的：第 ⑥ 行的倉庫網址，以及第 ④ 行的提交訊息。其餘原封不動照打即可。

🪜 點每一步，看它到底在做什麼

第一次看會覺得「怎麼這麼多指令」，其實它們是一條龍：建倉庫 → 擋垃圾 → 打包 → 存檔 → 命名 → 接遠端 → 推上去。點下面任一步，看白話解釋：

🧯 第一次最容易卡住的 3 個地方

這三個幾乎人人都中過一次。先有印象，真的遇到時就不會慌：

多半是開倉庫時不小心勾了 README / .gitignore，遠端已經有東西。先把它接下來，再推：

git pull --rebase origin main
git push -u origin main

GitHub 現在不能再用登入密碼推送。挑一條路：

方式怎麼做
Personal Access TokenGitHub → Settings → Developer settings → 產生 token，要密碼時把它當密碼貼上
SSH 金鑰（一勞永逸）用 ssh-keygen 產生金鑰、把公鑰貼到 GitHub，遠端網址改用 git@github.com:…

忘了第 ② 步。先把 .gitignore 補好，再把已經被追蹤的拿掉（本機檔案不會被刪）：

git rm -r --cached node_modules
git commit -m "Stop tracking node_modules"
🔑
如果交上去的是 .env 之類的密鑰，光移除還不夠 —— 一定要去把那把金鑰作廢換新。詳見「組合技 → 🆘 安全事故」。
🔗
這套是最精簡版本。想看含「遠端已有 README」「搬到新倉庫」等變化題，往下跳到 組合技 · 實戰食譜 🅱️。

核心觀念：先知道自己在做什麼

如果只記住一句話：Git 不是雲端硬碟，它是「幫你記住版本歷史」的工具。

👋 完全沒碰過 Git？先花 30 秒看這個

把 Git 想成一台幫你的專案不斷拍照存檔的相機。每完成一個段落就「拍一張」（commit），日後隨時能翻回任何一張舊照片。GitHub 則是把這些照片放上雲端、讓大家一起看的相簿。

📸commit拍一張存檔照
🌿branch另開一條平行劇情線
☁️push把照片上傳雲端
⬇️pull把雲端最新拿回來
🔁
整個 Git 旅程，其實就是不停重複這四步：
改檔案 → git add → git commit → git push。看不懂後面任何指令時，回到這句話就對了。

📖 名詞速查（看到不懂的詞就回來查）

這些詞會一直出現。不用背，先有印象，用久了自然記得。

詞白話解釋
repository / repo「倉庫」＝一個被 Git 管理的專案資料夾
commit一張存檔快照，有訊息、有時間、有專屬編號
branch（分支）一條平行的開發線，做新功能用，不影響主線
main主分支，通常是「正式、穩定」的那條線
HEAD「你現在站在哪一個版本」的箭頭
remote（遠端）雲端上的倉庫，例如 GitHub 上那一份
origin你的遠端的預設名字（慣例，不是魔法）
upstream「原作者」的倉庫；你 Fork 別人專案時會用到
clone把整個遠端倉庫複製到自己電腦
fork在 GitHub 上把別人的專案複製一份到你帳號
push / pull把本機提交推上遠端／把遠端最新拉回本機
merge（合併）把一條分支的內容接回另一條分支
PR（Pull Request）在 GitHub 上發起「請幫我把這些改動合併進去」的請求
staged（已暫存）已用 git add 挑好、準備這次提交的內容
.gitignore一份「這些檔案不要被 Git 追蹤」的清單

🗺️ Git 的四個核心區域

大多數指令，本質上都是在「把內容從一個區域搬到另一個區域」。點下面任一格看它的角色：

📝Working Tree工作區 →
📦Staging Area暫存區 →
🗄️Local Repo本機倉庫 →
☁️Remote Repo遠端倉庫

🔁 你之後最常做的基本流程

步驟指令目的
1git status看現在有哪些變更
2git add <file>把想提交的變更放進暫存區
3git commit -m "訊息"建立版本點（快照）
4git push推到遠端、和團隊同步

⚠️ 新手最常誤解的地方

❌ 常見誤解
✅ 正確理解
存檔就等於 Git 記住了
要到 commit 才真的形成版本
add 就是上傳
add 只是放進暫存區，沒上傳
Git＝GitHub
Git 是工具；GitHub 是託管平台
每次都要學很難的指令
先把高頻核心指令練熟最重要
🎯
本章重點：Git 的核心不是「備份」，而是「版本管理」。先學會看狀態 → 再學提交 → 效率會高很多。

第一次設定 Git 身分

你每次提交的作者名稱與 email，都是從這裡來的。安裝完 Git 後第一件該做的事。

⚙️ 一次設定好（可直接複製）

設定全域身分
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
git config --global --list
設定項目作用建議
user.name顯示在提交紀錄裡的作者名稱用真實或團隊慣用名稱
user.email提交紀錄裡的作者 email跟 GitHub 綁定信箱一致（頭像才會出現）
init.defaultBranchgit init 後的預設分支名建議直接用 main

🔍 --global 是什麼意思？

層級影響範圍什麼時候用
--global你電腦上所有 Git 專案個人預設值
不加 --global只有目前這一個專案公司專案需要不同 email 時
💡
某個專案想用不同身分，就在那個專案資料夾裡執行不加 --global 的 git config user.email "work@example.com" 覆蓋即可。
2

常用指令

每天都在用

這一組是 Git 的日常主力。把它們練到反射動作，你就已經會用 Git 了。

git init 建立版本庫

把一個普通資料夾變成 Git 專案。注意：init 只建倉庫骨架，不會自動 commit。

mkdir demo-repo && cd demo-repo
git init
git status
🆚
init vs clone：init 是自己建新倉庫；別人/團隊已有的專案則用 git clone（見「載別人」一節）。

git status 看懂目前狀態

Git 卡住時，第一個先打的指令，幾乎都應該是 git status。

git status
git status -s
狀態意思接下來通常做什麼
untracked新檔案還沒被追蹤git add <file>
modified已追蹤檔案被改了git diff 或 git add
staged已進暫存區，準備提交git commit -m "..."
nothing to commit沒有可提交變更可切分支、push 或繼續工作

git add 加入暫存區

git add 不是上傳，它是告訴 Git：「這些內容我要放進下一次提交。」

用法適合情境注意
git add <file>精準提交某個檔案，乾淨可控要多打一點字
git add .已確認所有變更都該提交容易把不想交的東西一起放進去
📸
add 加的是「當下內容的快照」。先 add 再修改，後面的修改不會自動跟進暫存區，要重新 add。

git commit 建立提交

沒有提交，就沒有真正的版本歷史。commit 才是 Git 記錄版本的時間點。

git commit -m "Add login form validation"

提交訊息怎麼寫最好 —— 一句公式：動詞 + 對象 + 目的

❌ 不好的寫法
✅ 較好的寫法
update
Update README introduction
fix
Fix login validation bug
change file
Rename settings page title
✨
最穩的提交流程：git status → git diff → git add → git commit -m。一次只做一件事，方便看歷史也方便回退。

git log 查看歷史

會看歷史，才算真的開始會用 Git。

git log --oneline

用簡短一行格式看提交歷史，最適合日常快速掃描。

git log --oneline --graph --decorate --all

用圖形化方式看所有分支怎麼分叉、怎麼合併。分支歷史一亂，這招最有幫助。

git diff 比較差異

養成「先 diff 再 commit」的習慣，能少踩很多坑（例如把除錯字串誤交上去）。

指令比較的對象什麼時候最有用
git diff工作區 vs 暫存區剛改完，想看還沒 add 的內容
git diff --staged暫存區 vs 最新提交確認這次提交到底會送出什麼
git diff HEAD工作區 vs 最新提交一次看全部本地變更
🔖
diff 裡 - 是被刪掉的內容、+ 是新增的內容。

git restore 還原修改

「我改壞了，能不能回到上一個乾淨狀態？」—— 通常可以。

指令影響範圍結果
git restore <file>工作區直接丟掉尚未提交的修改 執行前先 diff
git restore --staged <file>暫存區把檔案拿出暫存區，但保留工作區修改 較溫和
⚠️
git restore <file> 會丟掉未提交修改，且不易救回。執行前一定先 git diff 看清楚。

.gitignore 讓垃圾檔不要進 Git

編譯產物、快取、密碼檔，通常都不該提交。專案一開始就建立忽略規則。

.gitignore
node_modules/
dist/
.env
.vscode/
*.log
🧹
已經被追蹤的檔案，補寫 .gitignore 不會自動生效，要先取消追蹤：
git rm --cached .env（單檔）／ git rm -r --cached dist（整個資料夾）
--cached 只移除追蹤，不會刪掉你本機的檔案。

換向 · 切換分支

分支操作

分支的核心價值很簡單：把不同工作切開，讓主線 main 保持穩定。「換向」＝在分支之間穩定地切來切去。

git branch · git switch -c 建立分支

有新工作就先開分支，別直接在 main 上改。

git branch
git branch --show-current
git switch -c feature/login-page

switch -c＝建立新分支並立刻切過去，是非常高頻的日常指令。

類型命名範例
功能feature/login-page
修 bugfix/navbar-overlap
文件docs/update-readme

git switch 在分支間切換

切分支前先 git status，這是很值得養成的習慣。乾淨的工作區會讓切換輕鬆很多。

git switch main
git switch feature/login-page
git switch -

git switch - 可以快速「切回上一個分支」。

🤔
還不想提交、又要切走？用 git stash 先收起來（見「急救」一節），或先 commit 再切最穩。

git merge 合併分支

功能做完後，真正把它接回主線的動作就是 merge。

📍
一定要記住的規則：你站在哪裡，就把別人合進來。要把 feature/x 併入 main，就先切到 main 再 merge。

點「合併後」看 main 怎麼把 feature 接回來（橘色 HEAD 全程都在 main 上 —— 你是「接收方」）：

最常見的合併流程
git switch main
git merge feature/login-page
git branch -d feature/login-page   # 合併完刪掉本地分支，列表更乾淨
💥
遇到衝突？不可怕：① 打開衝突檔手動選要保留的內容 → ② git add <file> → ③ git commit 完成合併。詳見「急救與排錯」。

指向 · 版本移動

HEAD / reset / revert

Git 的歷史是一串提交，HEAD 是「你現在站在哪一個版本」的指標。「指向」＝移動這個指標，或讓分支指向別的版本。

🧠 先搞懂 HEAD 與「指向」

名詞白話解釋
HEAD「我現在站在哪個提交上」的箭頭
分支（branch）其實就是一個會跟著移動的「提交指標」
版本號（commit id）每筆提交的身分證，用 git log --oneline 查得到
git log --oneline

先用它查到你想去的版本號（前 7 碼即可），下面的指令才有目標。

🎯 HEAD 到底是什麼？點按鈕，看它怎麼移動

理解 HEAD 最快的方法：把它當成「你現在站在哪裡」的箭頭。依序點 1 → 5，看每個指令讓 HEAD（橘）和分支指標跑到哪裡：

💡
一句話記住：commit 是「拍好的照片」，分支是貼在某張照片上的便利貼，HEAD 是「你正盯著哪張便利貼」。移動 HEAD ＝ 換你站的位置。

git switch 指到指定版本

想「回去看某個舊版本長怎樣」，或從某個舊版本「另開一條新分支」：

指令用途
git switch --detach <版本號>把 HEAD 暫時指到某個舊版本（進入 detached 狀態，只看不改）
git switch -c <分支名> <版本號>從指定版本出發建立新分支並切過去（保留工作最安全）
git switch --detach 3f6181b
git switch -c fix/from-old 3f6181b
🧭
Detached HEAD（分離頭指標）：你不在正式分支上，只是停在某個提交。看完想回去就 git switch main；如果在這裡做了想保留的修改，立刻 git switch -c rescue/xxx 開新分支接住它。

git reset 把指標往回拉（未 push）

把分支指標往回移，撤銷 add 或 commit。三種模式差在「動到哪幾層」—— 點按鈕看每一層是保留還是清掉：

📋 完整對照表：

指令commit暫存區工作區白話
git reset --soft HEAD~1退回保留 staged不動「這筆 commit 想重做，內容我還要」 低
git reset HEAD~1退回重設修改留著「拆掉重整，內容放回工作區」 中
git reset --hard HEAD~1退回清掉一起退「這整段都不要了」 高
🔥
--hard 會把本機修改一起清掉，且難救回。只要你還想保留內容，就先不要碰 --hard。退錯了第一時間看 git reflog。

git revert 安全撤銷（已 push 首選）

新手最需要的不是最猛的破壞性指令，而是最安全的回退方式。

git revert <commit>

revert 不刪歷史，而是新增一筆「反向提交」把舊提交抵消掉，因此在已分享/共享分支上最穩。

⚖️
reset vs revert：reset 會改寫歷史（適合還沒 push）；revert 不改歷史（適合已經 push）。

git reflog 救援地圖

記錄「你的 HEAD 最近去過哪裡」—— 連 reset、switch、amend 之後找不到的位置都常能找回來。

git reflog
git reset --hard HEAD@{1}
🛟
剛剛 reset 過頭、馬上後悔 → 先 git reflog 看 HEAD 軌跡，找到剛剛那個位置再 git reset --hard HEAD@{1} 切回去。先別判定「完了」。

載別人 · 取得專案

clone / fetch / pull

如果專案已經存在於遠端（GitHub 等），通常不是 init，而是直接「載下來」。

git clone 複製整個專案

把遠端專案完整複製到本機（含所有歷史）。這是拿到別人專案最常見的第一步。

clone 別人的專案
git clone https://github.com/example/project.git
cd project
git remote -v   # 看遠端連到哪

git remote 管理遠端連線

如果你是本地先做、現在才要接上遠端：

git remote add origin https://github.com/example/project.git
git remote -v
🏷️
origin 只是遠端倉庫的預設慣例名稱，不是特殊魔法。幾乎所有教學都這樣叫。

git fetch · git pull 拉回最新

協作時，不只要會推，也要會穩定地把別人的更新拉回來。

指令做了什麼適合情境
git fetch origin只更新遠端資訊，不改你目前工作內容你想先觀察 保守
git pull --rebase origin main抓下來，並把你的本地提交接到最新遠端之後你準備正式同步
✨
為什麼很多人愛 pull --rebase：歷史較乾淨，不會多出一堆不必要的 merge commit；先把遠端最新接上，再放回你的提交。同步前先確認工作區乾淨，會少很多麻煩。

跟別人合作 GitHub

push / pull / 協作流程

你本機 commit 完，團隊還看不到；要 push 才會同步到遠端。這一節把「和別人一起開發」串成完整流程。

git push 把版本推上遠端

場景指令
第一次推主分支git push -u origin main
新功能分支第一次上傳git push -u origin feature/login-page
同一分支後續更新git push
🔗
-u（＝--set-upstream）設定 upstream：之後這個本地分支就知道自己對應哪個遠端分支，日常多半只要打 git push。

🔄 標準協作流程（最重要）

和別人共用一個 repo 時，一個功能從頭到尾的節奏：

同步主線
git switch main && git pull --rebase origin main
開新分支做事
git switch -c feature/navbar-redesign
提交並推上遠端
git add . && git commit -m "Redesign navbar" && git push -u origin feature/navbar-redesign
在 GitHub 開 Pull Request，請人 review、合併團隊協作通常透過 PR 合併，而不是自己直接 push 到 main。
合併後回本機同步、刪分支
git switch main && git pull --rebase origin main && git branch -d feature/navbar-redesign

🚧 協作時最常撞到的兩個狀況

遠端已經有比你更新的內容，不能直接硬推蓋掉。先把別人的更新拉下來，再推：

先拉再推
git fetch origin
git pull --rebase origin main
git push

同一段內容被不同人改成不同版本，Git 不敢替你決定。打開衝突檔，你會看到：

衝突標記
<<<<<<< HEAD
你的版本
=======
對方版本
>>>>>>> feature/x

手動改成最終內容、刪掉 <<< === >>> 標記，存檔後告訴 Git 處理完了：

git add app.js
git commit # 一般 merge
git rebase --continue # 若是 pull --rebase
🧩
衝突不是叫你二選一。很多時候正確答案是把兩邊重新整理成第三種版本。

🛡️ 改到別人也在用的歷史時：請手下留情

分支類型安全原則
你自己的功能分支可考慮改歷史，但仍要小心；強推用 git push --force-with-lease（比 --force 安全）
團隊共用的 main／共享分支優先用 git revert，不要直接重寫歷史或 force push
⛔
看到 push --force 先停一下，問自己：這是不是共享分支？幾乎所有共享分支都不該 force push。

組合技 · 實戰食譜

照著做就會

單一指令學會後，真正的工作其實是「好幾個指令串起來」。這一節把最常見的實戰情境整理成食譜 —— 跟著步驟做就好。

🅰️ 把別人的作品弄下來

git 別人的 GitHub 專案

「我想用 / 研究 / 參與別人的專案」—— 依你的目的選一種：

最單純：把整個專案連同歷史複製到本機。

clone 下來
git clone https://github.com/某人/專案.git
cd 專案
⚡
專案很大、只想看最新版？加 --depth 1 只抓最近一筆，下載快很多：git clone --depth 1 <url>

你想用別人的程式碼當起點，但不要他的版本歷史，要從你自己的第一筆 commit 開始：

下載 → 清掉舊歷史 → 重新開始
git clone https://github.com/某人/template.git my-project
cd my-project
rm -rf .git              # 移除原作者的版本歷史
git init                  # 重新開一份屬於你的歷史
git add .
git commit -m "Initial commit from template"
🪟
Windows PowerShell 沒有 rm -rf，改用 Remove-Item -Recurse -Force .git（Git Bash 則可直接用 rm -rf）。

要把修改交回給原作者，標準做法是「Fork」。

在 GitHub 網站點 Fork把專案複製一份到你自己的帳號底下。
clone 你自己的 fork（不是原作者的）
git clone https://github.com/你的帳號/專案.git && cd 專案
把原作者倉庫設成 upstream之後才好把原作者的更新同步回來。
git remote add upstream https://github.com/原作者/專案.git
開分支、改東西、提交
git switch -c fix/typo && git add . && git commit -m "Fix typo in README"
推到你的 fork
git push -u origin fix/typo
回 GitHub 點「Compare & pull request」向原作者發 PR，等他 review、合併。

原作者後來又更新了，你想把那些更新同步進自己的 fork（需先做過 A3 設定 upstream）：

同步原作者更新
git fetch upstream
git switch main
git merge upstream/main   # 把原作者的 main 併進來
git push                   # 把同步後的結果推回你的 fork

🔺 origin / upstream / 你的電腦：一張圖搞懂三者

「貢獻別人專案」最容易暈的就是這三個倉庫的關係。點 ① → ⑤，看每一步資料往哪邊流：

🧭
一句話分辨：origin ＝「我的那份（fork）」，upstream ＝「原作者那份」。你 push 永遠進 origin；要回饋原作者就發 PR。
🅱️ 把自己的作品推到倉庫

把現在的作品 git 上去 / 搬到新倉庫

本機已經有一個專案，想讓它「上雲端」或「換一個新家」：

先在 GitHub 建一個「空的」新倉庫先別勾 README / .gitignore，免得和你本機內容衝突。
本機初始化並提交
git init && git add . && git commit -m "Initial commit"
接上遠端並推上去
接遠端 + 首推
git branch -M main                          # 主分支命名為 main
git remote add origin https://github.com/你的帳號/新倉庫.git
git push -u origin main
🛑
如果 push 被擋（遠端已有 README）：先 git pull --rebase origin main 把它接下來，再 git push。

專案原本連著舊倉庫，現在想改推到一個新倉庫。先看目前連到哪：

git remote -v

直接把 origin 改指向新網址，再推：

換掉 origin 指向
git remote set-url origin https://github.com/你的帳號/新倉庫.git
git remote -v                    # 確認改成功
git push -u origin main
💡
想同時保留舊倉庫、另外加一個新的，就用 git remote add new <新網址>，再 git push new main。

不只搬 main，而是整個倉庫（所有分支、所有 tag）一字不漏搬到新倉庫。用 --mirror 最乾淨：

整庫鏡像搬移
git clone --bare https://github.com/舊/專案.git
cd 專案.git
git push --mirror https://github.com/新/專案.git
多加一個遠端
git remote add backup https://github.com/你的帳號/備份倉庫.git
git push backup main

之後 git push 推 origin、git push backup 推備份，各推各的。

🛠️ 日常開發常見組合

寫程式時三天兩頭會用到的串接

這些不是「出事了」才用，而是正常開發的日常節奏。挑你現在遇到的展開來看：

情況一：還沒 commit —— 最簡單，直接帶著修改開新分支：

git switch -c feature/right-branch

未提交的修改會跟著你到新分支，原地繼續做即可，main 完全沒被動到。

情況二：已經 commit 在 main 上（還沒 push） —— 先用新分支接住，再把 main 退回乾淨：

把 commit 搬到正確分支
git switch -c feature/right-branch   # 新分支接住剛剛的 commit
git switch main
git reset --hard origin/main          # 把 main 拉回和遠端一樣
⚠️
reset --hard 會丟掉 main 上多出來的提交 —— 但它們已經安全待在新分支了才這樣做。退錯了第一時間 git reflog 找回來。

半成品還不想 commit，但要你馬上修個急 bug：

stash 救火流程
git stash -u            # 連未追蹤的新檔一起收起來
git switch main        # 切去處理急事…改完、commit、push
git switch -           # 切回剛剛那個分支
git stash pop          # 把半成品拿回來繼續做
📌
git stash list 可看收了哪些；pop 若遇衝突，stash 有時不會自動刪掉，記得再確認一次。

同一個檔案裡，有些改動想這次交、有些想留到下次。用 -p 一段一段挑：

git add -p

Git 會一塊一塊問你要不要加，常用回應：y 要、n 不要、s 拆更細、q 結束。挑完照常提交：

git commit -m "Add only the validation part"
🎯
養成「一個 commit 只做一件事」的習慣，日後看歷史、回退都會輕鬆非常多。

不想整條分支合併，只要那一筆修正 —— 用 cherry-pick：

撿單一 commit
git log --oneline other-branch   # 找到要的那筆 commit id
git switch main
git cherry-pick a1b2c3d            # 只把那一筆套到目前分支

功能告一段落，想標記「這就是 v1.0.0」：

打標籤 + 推標籤
git tag -a v1.0.0 -m "First release"
git push origin v1.0.0        # 單獨推這個 tag
git push origin --tags        # 或一次推全部 tag
💡
git push 預設不會把 tag 一起推，tag 要像上面這樣自己單獨推。

只想讓單一檔案回到某個歷史版本，其他檔案都不動：

還原單一檔案到舊版本
git log --oneline -- src/app.js          # 找到那個版本的 id
git restore --source a1b2c3d -- src/app.js
🔎
還原後檔案會變成舊版內容、留在工作區，確認沒問題再 git add + git commit 即可。
🆘 安全事故

不小心把金鑰 / 密碼 push 上去了

🚨
第一件事，不是刪檔，是「作廢金鑰」。
GitHub 上的密鑰，往往幾秒內就被掃描機器人撈走；而且任何已經 clone / fork 的人手上都還有那份。所以唯一真正有效的補救是：立刻把那把金鑰／密碼作廢、重新產生一把新的（rotate）。下面的清歷史只是收尾，不能取代這一步。
移除 + 加入 .gitignore + 重新提交
git rm --cached .env                 # 停止追蹤，本機檔案保留
echo ".env" >> .gitignore             # 以後不再追蹤它
git commit -m "Remove leaked secret, ignore .env"
git push
⚠️
這只移除「最新版本」。舊的提交歷史裡那把金鑰還看得到 —— 所以仍要做 C2 清歷史，並且務必已經作廢金鑰。

官方推薦 git filter-repo（需另外安裝），把某個檔案從所有歷史抹掉：

git filter-repo --invert-paths --path .env

或用 BFG Repo-Cleaner：

BFG 清檔 + 收尾
bfg --delete-files .env
git reflog expire --expire=now --all
git gc --prune=now --aggressive

歷史已被改寫，需要強推，並通知所有協作者重新 clone：

git push --force --all && git push --force --tags
🔁
再強調一次：清完歷史不代表安全。已外洩的金鑰一定要作廢換新，否則別人手上的舊副本仍然有效。
做法說明
一開始就寫好 .gitignore把 .env、*.key、id_rsa、憑證檔先擋掉
密鑰放環境變數不要把金鑰寫死在程式碼裡
開啟 GitHub Secret scanning / Push protection推之前就幫你攔下疑似密鑰
提交前 git diff --staged養成習慣，提交前看一眼到底交了什麼

急救與排錯

手滑補救

真正常見的不是你不會 Git，而是你手太快。先判斷錯在哪一層，再選最安全的補救方式。

git stash 暫時收起未完成工作

做到一半突然要切去修別的東西，又不想為半成品硬 commit —— stash 超好用。

指令用途
git stash暫時收起目前修改
git stash -u連未追蹤的新檔案一起收起來
git stash list看有哪些 stash 紀錄
git stash pop把最近一筆 stash 套回來並移除
📌
stash pop 若發生衝突，stash 有時不會被刪掉。別以為它一定消失了，再 git stash list 確認一下。

git commit --amend 修正最近一次提交

不是要回退整個歷史，只是上一筆 commit 少放了檔案、或訊息打錯。

git add forgotten-file.txt && git commit --amend --no-edit
git commit --amend -m "Correct commit message"
⚠️
amend 本質上是重做上一筆。還沒 push 最安全；已經 push 且團隊可能拉走的歷史，先別隨便 amend。

🚑 25 個最常見「手滑」情境（點開展開）

先判斷：你卡在檔案、暫存區、commit、還是遠端？再對症下藥。

改壞 / 誤刪單一檔案，想回到目前 commit 版本：

git restore main.c

整個工作區改亂，全部恢復：

git restore .

只想救某個資料夾：

git restore src/
🚫
本地刪檔想救回時，先別用 git pull。遠端若沒有新 commit，pull 不會幫你補回本地刪掉的檔案，用 git restore 才對。

單檔 add 錯，取消 staged（保留修改）：

git restore --staged secret.txt

git add . 加太多，全部取消 staged：

git restore --staged .

連工作區修改也不要：

git restore --staged config.yml && git restore config.yml

訊息寫錯：

git commit --amend -m "Correct commit message"

少加一個檔案：

git add missing-file.txt && git commit --amend --no-edit

撤銷最後一次 commit，但保留修改在工作區：

git reset HEAD~1

撤銷最後一次 commit，連修改都不要： 高風險

git reset --hard HEAD~1

安全撤銷（共享分支首選，不改歷史）：

git revert HEAD

撤銷某個舊 commit：

git revert <commit_id>

確定是自己的私人分支、可改遠端歷史： 高風險

git reset --hard HEAD~1 && git push --force-with-lease

只是想抓遠端最新：

git pull

push 被拒（遠端比你新）：

git pull --rebase && git push

讓本機完全變成 GitHub 那版： 高風險

先試跑 clean -nd 再真刪
git fetch origin
git reset --hard origin/main
git clean -nd   # 只列出會刪什麼，不真刪
git clean -fd   # 確認後才真的刪未追蹤檔

剛剛 reset 過頭，想找回位置：

git reflog
git reset --hard HEAD@{1}

🚦 危險操作前：固定先問 4 個問題

#先問自己影響
1要取消的是哪一層？（工作區／暫存區／commit／遠端）決定用哪個指令
2那筆提交有沒有 push 過？沒 push 可較自由整理；已 push 要小心
3這是共享分支嗎？共享分支優先 revert
4現在有沒有未提交修改？先別把自己的東西弄丟
🦺
怕退錯就先做保險：① 開備份分支 git branch backup/before-recovery　② 先收修改 git stash -u　③ 刪未追蹤檔前先試跑 git clean -nd。

🧯 卡住情境快速判斷

你看到的情況常見原因優先做法
切 branch 被擋本機有未提交修改commit / stash / restore
pull 被擋工作區不乾淨先清工作區，再 pull --rebase
push 被拒遠端比你新先 fetch / pull --rebase 再 push
merge / stash pop 衝突同段內容被改成不同版本手動解 → git add → 完成流程
branch 刪不掉Git 判定還沒合併確認後 -d（已併）或 -D（強刪）
push 說沒有 upstream分支還沒綁定遠端git push -u origin <branch>
Detached HEAD切到某個 commit，不在正式分支切回 branch，或立刻開新 branch 保留工作
🧘
Git 卡住時，先看 git status，不要先慌。很多問題不是 Git 壞掉，而是它在保護你不要覆蓋本機或遠端內容。

每日 Git 工作流總複習

把學過的高頻指令串成一套流程，這就是大多數人每天在做的事。

階段核心指令
同步主線git switch main、git pull --rebase origin main
開新工作git switch -c feature/...
檢查與提交git status、git diff、git add、git commit -m
推上遠端git push -u origin <branch> 或 git push
合併收尾git switch main、git merge <branch>、git branch -d
🏁
五條實戰心法：① 卡住先 status　② 提交前先 diff　③ 每個功能開新分支，別硬在 main 改　④ 已推送的歷史優先 revert，別衝動危險重寫　⑤ 不知道剛剛發生什麼，就看 log 或 reflog。

🧠 三組最容易搞混的差別

對照關鍵差別
restore vs resetrestore 還原檔案內容／取消 staged；reset 把Git 狀態（commit）往回拉
reset vs revertreset 會改歷史（未 push）；revert 不改歷史（已 push）
pull vs restorepull 抓遠端最新；restore 從本地 commit 還原檔案（不是抓遠端）

📊 風險分級（越往下越危險）

指令風險為什麼
git status / log / diff很安全只看，不改任何內容
git stash / git revert相對安全先收起來 / 新增反向提交，可回復性高
git reset --soft / git reset中風險退掉 commit，但修改還在
git reset --hard / git clean -fd高風險直接刪掉本機內容
git push --force很高風險可能改亂遠端共享歷史
🧭
方向感原則：越會「改寫歷史」、越會「直接刪本機」、越會「影響遠端與隊友」的指令 —— 通常越危險。動手前先看 status、log、diff。
